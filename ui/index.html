Risk Decision Engine Admin

Welcome to the Risk Engine Admin

Overview

This application manages a data driven Risk Decision Engine. It allows you to:

  • Configure Tenants: Different organizations or lines of business.
  • Set up Checkpoints: Each checkpoint is a decision point, with an associated DSL expression describing how signals combine to produce a final result.
  • Register and manage Signals: The data retrieval or computation operations (e.g. internal/external APIs, local functions, etc.).
  • Test your checkpoints using the Test Checkpoints tab.
  • Audit and Search logs of decisions, signals, and more.

DSL Authoring Guide (Extended)

The dsl_expression for each checkpoint is evaluated using the SimpleEval library, which is a safe subset of Python syntax. You can think of it as writing small Python-like expressions without needing the full overhead of a Python environment where all configured signals become variables in the DSL. For instance, if your checkpoint has signals named age_check, blocklist_check, and kyc_score, you can write expressions such as: 

age_check and not blocklist_check and (kyc_score > 80)

After the signals are evaluated, they populate the DSL environment. Booleans, numbers, or strings can be checked or compared. Typically, the final DSL result should be True or False to indicate "Approve" vs. "Deny," or "Pass" vs. "Fail."

Allowed Operations

  • Logical operators: and, or, not
  • Comparison: ==, !=, <, <=, >, >=
  • Arithmetic: +, -, *, /, //, **, and % (modulo)
  • Parentheses to group operations: ( ... )
  • Boolean literals: True, False
  • Functions: max(), min(), abs(), round(), sum(), and len()
  • String literals: 'some text' or "some text"

Examples

If you have a numeric signal credit_score and a boolean signal eligible, you might write: 

(credit_score >= 70) and eligible

This expression returns True if credit_score is at least 70 and eligible is true, otherwise False.

Or, if you have account_balance (number) and days_overdue (number), you could do: 

(account_balance > 1000) and (days_overdue <= 5)

Working with Strings

If a signal is a string, you can compare it using ==, !=, etc. For example: 

region == "US"

Built-in Functions

You can use common functions such as max(), min(), abs(), round(), sum(), and len() directly in your DSL expressions. For example: 

max(10, credit_score)

This returns whichever value is larger (10 or credit_score). Note that any other function calls not on our allow list will be blocked by default unless added.

Placeholders inside Signals

Some signals have placeholders in their templates (e.g. %user_ssn%). You set those before evaluation. Once the signal is called, its result is placed into your DSL environment. That means if your signal name is kyc_score, then inside the DSL, you see something like kyc_score == 90. The placeholders used by the signal do not directly appear in the DSL, they're resolved first by the engine.

Tips & Best Practices

  • Use parentheses liberally for clarity: ( credit_score > 70 ) and ( not blocklist_check ).
  • Keep your expressions short and simple and combine signals in an intuitive way rather than nesting too deeply.
  • Test in the "Test Decisions" tab, see if the final evaluation matches your expectations.
